抢购期间出现服务本地端口资源耗尽的报错问题

问题影响

服务高峰期间服务通过rpc请求另一个服务，偶现出现少量下方报错，显示本地端口资源耗尽：

问题分析

1、 观察单机接口的qps量级并不是历史最高，突然出现这个错误怀疑是环境资源被调整引起，由运维继续追查环境问题端口池配置的范围，这里只关注研发的代码优化
2、 检查代码中网络请求逻辑，发现函数中直接使用resty.New()创建大量临时实例，这样导致连接不能被复用，高峰期出现大量tcp time_wait的连接
3、 由于代码中都是短连接的交互形式，出现端口资源耗尽可能是下游耗时异常导致的瞬时持有连接数量过高，或者占用的端口回收不及时如TIME_WAIT类的连接，这里完善下游接口的耗时日志，辅助判断问题

解决方案

1、 resty.New()的client实例维护成全局单例
2、 设置resty的transport，MaxIdleConns和MaxIdleConnsPerHost属性合理配置连接池，加强复用率
3、 调整失败重试的间隔，默认的100ms并不适合这个场景，只会无意义的加重问题，根据业务情况适当延长，分散高峰点附近的请求密度

抢购开始的前几秒，热点业务链路的接口压力过大，waf网关qps限制被打满

问题影响

周四晚上的爆款抢购由于业务规模增长的速度超出预期，开抢前几秒系统压力就过高，扩容已经来不及，导致一个mysql实例cpu打满，一个redis实例带宽流量打满，接口耗时异常增长超时，用户页面卡顿乃至无法加载出来
另一方面前几分钟的火爆程度较高，waf网关的20w qps被持续打满，导致部分用户在页面上时不时出现网关超限的网络报错，体验感较差

解决方案

问题点比较多，隔天下来之后与测试部门联合做了链路的压测，逐步排查出压力高的环节，逐个进行优化，具体的优化方向如下：

  • 1、部分非关键模块进行降级
    高峰期抢购期间，商详页的一些周边逻辑给系统带来了较大压力影响，这些模块很多设计的时候并没有支持高并发的能力，mysql的cpu飙高有一部分原因来自于此，这部分非主要链路的功能入口决定在高峰期期间直接进行隐藏，避免为这些周围逻辑消耗大量精力优化
  • 2、本地缓存优化
    部分高并发的接口实现较为粗糙，一些商品信息之类的偏静态数据主要靠redis来抗查询压力，此部分改为加入服务器本地缓存，以服务器静态文件+短expire的内存local cache的分担redis流量压力
    缓存中排查时发现还存在部分大key，频繁的拉取占用了大量网络资源，业务上拆分大key，静态数据和动态数据分离，配合本地缓存降低对redis的网络交互流量
  • 3、接口整流
    类似商品分页列表/商品详情/活动详情/相关配置等等偏静态数据读取的接口，加入single light整合请求流量，降低处理压力
  • 4、waf网关部分接口分割出去
    接口经过waf主要是为了防止一些恶意请求，限流和做规则过滤，这里不需要所有接口都经过waf，将不需要处理的高频接口迁出waf，直接分散到单独的服务入口
  • 5、异步逻辑放入mq
    业务逻辑中有很多抛出goroutine的异步逻辑，高qps下会抛出过多的协程，产生不可控的影响，量级较大的地方都丢到mq中由固定的业务线程组消费处理
  • 6、代码细节优化
    针对压测过程中耗时较久的个别接口专门优化实现，使用errgroup能并发处理的地方并发加速，替换sonic等更高效的序列化/反序列化方法，一些逻辑结果缓存到本地内存复用，可以做去重逻辑的位置使用内存记录做短时间的幂等过滤

抢购用户数量超出预期，导致微信网关带宽流量超出限制

问题影响

记录一次线上流量激增带来的问题，商品热度高于预期，参与抢购的用户数量比以往高了10倍以上，导致微信网关瞬时流量超过5G，部分用户由于请求不通页面异常无法下单

排查方法

1、从日志服务alb日志和网关日志中根据请求接口的路由进行分组统计，alb和网关日志一般都会记录请求/响应的字节大小，根据接口路由统计后捞出 接口的请求次数/接口request平均大小/接口response平均大小/一定时段内总流量大小（一般拉取出问题期间的一分钟的数据即可）
2、寻找对照组，将上一次同样售卖爆款的正常场景也同样进行数据统计，比对两场售卖的接口数据差异，找到变化最大的一部分接口，分析其情况是否正常，因为很多流量高峰其实是黑产刷接口导致的，根据业务实际情况判断差异的根因进行解决
3、对步骤一的日志按接口总流量大小的占比进行排序，找到占比最大的一批接口，对其进行分类判断（大数据高频率接口/小数据高频率接口/不常变化的静态属性接口）
4、与前端/产品配合逐个分析头部接口，结合分类确定优化方向

解决方案

一般可根据下面几种情况对接口进行分类来优化：

  • 单次接口调用的平均大小（平均请求+平均响应）如果较大的话（例如超过5kb），即属于大数据接口，这种一般是响应数据较大，优化思路为删减字段/拆分接口；需要分析前端配合梳理响应字段的使用情况，干掉无用的字段，部分通用的接口会存在几部分模块数据都在一个接口里返回，这种情况下可以根据业务拆解这个接口，单个前端场景只使用需要的字段
  • 一些短时间数据不会变化的接口，例如spu/sku的一些静态属性接口，其实不需要频繁拉取，用户在切换页面时可能会频繁触发调用，这种情况可以考虑前端做缓存或后端配合做304code策略（后端对每次接口响应做hash，将hash值在响应数据中返回，前端缓存同一接口的上次响应和hash，每次调用时在header中传入hash，后端判断到响应数据没有变化时，将http code置为304，这样就不需要返回响应数据，在response比较大的情况下效果会很好）
  • 单次接口调用的平均大小比较小的情况，这种一般是因为频率极高导致占用的带宽也很大，优化思路为减少接口调用频率
    除了针对场景的优化，还有一些通用的优化方法：
  • 实行降级策略，后端控制降级开关，在高峰期暂时隐藏部分不重要的功能，较为适用于定时发售抢购的场景，可以大幅降低系统压力
  • 响应体大的接口如果无法删减字段，可以考虑采用压缩算法对响应体进行压缩，减少传输的带宽占用
  • 以上优化执行后还是无法得到较好的效果，可以考虑拆分服务集群分开部署，物理上横行扩展分散网关流量

并发读写panic问题

比较基础的go并发问题，map应配合锁或chan等其他机制避免并发读写的碰撞

map受到流量冲击后内存暴增问题

很多有时效要求，高并发的场景下会选择将数据缓存到内存的map中，在一些数据动态增删的场景下会出现服务内存使用量走高之后无法下降的情况，其中一种可能的情况是全局map高峰期扩容后，数据虽然delete掉但内存没有释放掉，即map内存泄露
常见解决办法为：

  1. 周期性更新全局map，创建新的map，将旧map有效元素拷贝入新map，新map内都是有效数据，然后将旧map指针指向新map，旧空间等待gc回收，还能避免频繁删除造成的map overflow链表碎片化问题
  2. 注意map元素的类型，若单个元素很大，优先使用指针类型进行存储，delete时删除引用关系，元素实际数据也可以后续被gc回收，也能起到实时减少内存的效果（注意使用指针会加大gc扫描标记的消耗，得根据实际情况（单struct较大/频繁新增删除/指针对象不会被其他地方引用）取舍是否要用这种方法）

map键类型为指针问题

map的键类型语法上只要求可以比较是否相等即可，所有指针类型也可以写为key并不会触发语法错误，但是这样会导致代码复杂且不可控应避免这种写法
指针作为map的键时可能出现的问题：

  1. 指针指向的对象发生扩容迁移等内存重新分配时，指针地址可能变化，map键中存储的旧地址值不会改变，将导致在map中无法比较成功
  2. 插入数据时无法比较成功导致内存持续上涨，实际运行场景中因为内存频繁变更，map中会出现大量无法比较成功的悬空key，这些指针本身的空间和引用的对象实体都无法得到释放，除非重建map，内存占用会随着map不断插入数据而不断膨胀

map遍历读取的随机性分布并不均匀

go的map遍历读取元素时顺序是随机的，根本原因是map底层hashmap扩容迁移等操作都会导致顺序不一致，为了保证特性不被误用，go做了随机化处理，即使静态map没有扩容每次遍历的顺序也是随机的，但是这个随机的分布是不保证均匀的，不应该被用做业务里的随机化操作

不常变动的大量级map使用非指针类型加速gc

大map在程序运行时会影响程序的gc性能，当量级超过千万级别，影响就相当明显，若map的key/value类型都是非指针类型，且大小不超过128字节的时候，gc时可以避免指针追踪带来的递归扫描并使用小对象的更高效扫描策略

访问控制casbin

casbin是一种开源的访问控制框架，设计目标是提供一种简单、灵活且易于扩展的访问控制解决方案，支持多种访问控制模型，下面是基于rbac角色权限控制的方式实现的一套系统实践。
官方文档：https://casbin.org/zh/docs/rbac/

核心概念

  • 模型（Model）：
    Casbin 使用一个配置文件（通常是 .conf 格式）来定义访问控制模型。模型文件中定义了各种规则，如用户、角色、资源、权限等之间的关系。例如，在 RBAC 模型中，模型文件会定义角色与用户的关联，以及角色与权限的关联。
  • 策略（Policy）：
    策略是一组具体的访问控制规则，它定义了哪些用户或角色可以对哪些资源执行哪些操作。策略通常以表格的形式存储，可以通过 Casbin 的 API 进行增删改查操作。
  • 请求（Request）：
    在应用程序中，当用户尝试访问某个资源时，会生成一个请求。请求包含了用户信息、操作信息和资源信息。Casbin 根据模型和策略来判断该请求是否被允许。
  • 适配器（Adapter）：
    适配器用于将策略数据存储到不同的数据源中，如内存、文件、数据库等。Casbin 提供了多种适配器。

    这部分定义了用户传入的数据如何与规则进行匹配，类似检索条件。

    3、r.sub == "0"表示如果 sub 等于 “0”，无论如何都允许请求通过

redis使用遇到的一些问题记录

redis大key问题

问题影响

大key就像手捧雷，兜兜转转，随机折磨一个倒霉蛋。。。。

  • 阻塞问题/慢查询：
    对大Key的读、写、删除操作时间较长，可能会阻塞Redis主线程，导致服务不可用，并且主从redis场景下可能导致同步数据的延迟变高
  • 网络延迟：
    传输大Key的数据会增加网络 I/O 延迟
  • 分片存储不均：
    redis集群版的场景下，大key数据一般都会都落到一个分片上，导致数据倾斜
  • 删除耗时异常：
    部分更新场景可能清掉大key重新设置，这个操作可能造成O(m)的时间复杂度和大量内存块的同步释放，可能造成阻塞
排查方法
  • 使用 Redis CLI 工具扫描大Key：redis-cli --bigkeys
  • 慢日志（slowlog）分析：redis-cli slowlog get 10
解决思路
  • 合理拆分大Key：
    业务设计时应该充分评估一个缓存的量级，选择正确的数据结构和映射方式。
    针对列表、集合、哈希表等：将大Key按业务逻辑拆成多个小Key，例如一个用户订单列表可以按年月拆分为多个列表，用户属性可以分类拆到多个小Key
  • 分页操作：
    对于列表、集合和有序集合的查询，避免一次性读取大量数据，使用 LRANGE、SSCAN、ZSCAN 等支持分页的命令
  • 异步删除大Key：
    避免直接删除大Key导致阻塞，应使用UNLINK命令异步删除数据，或业务自己分批实现小量级的删除。

多级缓存失效问题

在应用localCache配合redis做多级缓存的场景下，需要注意key的过期时间，避免流量高峰期大批量的本地缓存失效，具体操作起来可以参考：

  1. 可对localCache的key过期时间做随机的打散
  2. 类似节假日主推的活动这类静态只读的热点数据可以常驻在内存中，或者命中本地缓存时直接续期本地缓存做滑动过期
  3. 本地缓存失效去redis拉取数据时，可以使用golang.org/x/sync/singleflight包做单机纬度的请求整合，防止瞬间的击穿，一些组合逻辑复杂的静态分页场景也可以使用这个库

多级缓存数据一致性问题

系统引入localCache后需要注意使用的场景和内存里缓存的时效，在redis数据发生变化时，位于容器或ecs的集群服务无法快速感知到，这就会出现短暂的localCache和redis数据不一致，这种情况的解决办法可以参考：

  1. localCache的缓存时间较长并可以接受短暂的数据不一致，可以使用redis的订阅能力，redis数据更新后发布消息，集群服务收到消息后更新localCache
  2. 若要求延迟低的场景，内存缓存时间设置的短一些，也可以较为简单快速更新localCache，但是这样redis的压力会增加，需要根据业务场景进行权衡
  3. 另一种方案是内存缓存自主的定期更新，比如设置内存缓存key生命周期为10s，由框架或者其他协程按每s一次的频率从redis更新这些key，适用于key数量不高的情况，将redis访问的被动不可控频率变为主动的可控频率

集群式redis问题

业务达到一定量级之后，单节点redis的cpu能力/网络连接数能力/网络带宽等都可能不足以支撑业务，此时就需要分散压力，使用多分片的集群式redis，一些问题也会随之出现

  • lua/txPipeline操作多个key时的限制：
    多分片redis在处理批量命令的lua脚本/事务时，涉及操作的所有key必须保证在一个分片上，一般我们都会使用hashtag的方式来控制这些key的slot位置，即key命名时将想要参与hash的部分字符串用{}括起，redis会根据括号内的字符串进行hash计算，这就需要业务在设计缓存key的初期对拓展有所预判，若判断多个key存在联动操作的可能，就应该合理规划其命名
    使用hashtag时需要注意均匀slot的数据分布，{}中间的字符串应尽量包括类似id这样适合hash分散数据的内容，否则会人为的造成数据倾斜，redis可用性降低
  • 分片压力不均：
    由于可能存在热点key，导致某几个分片的压力特别高
    此时需要先判断热点Key的实际使用场景，若是简单的只读场景，则可以考虑引入localCache抗量或考虑拆分热点key业务逻辑上分解数据，压力分担到其他分片，若是读写混合场景，则可以考虑将热点模块迁移到独立的主从redis，将问题分割出去单独扩展资源
  • 小心大key：
    集群redis场景一般都会涉及扩缩容，大key在这个场景下迁移会耗时较长，可能阻塞redis导致业务出现问题，尽量避免使用大key，若必须使用大key，则尽量避免使用lua脚本

key命名问题

redis的key应尽量用独特的业务标识缩写用:进行拼接，避免一些存在共性和无意义的字符串，且不宜过长避免占用过多内存，若要用到较长内容作为key时可以考虑使用md5等方式将key统一为较短字符串
同一业务内应维护一套统一的命名规范，命名中可以体现出redis的数据结构，方便使用者分辨理解

主从redis问题

  • 机器系统时间不一致：
    主从节点所在集群本地时间不一致就会导致key的过期时间判断出现问题
  • 脑裂问题：
    redis的哨兵机制会导致主从身份的变化，当集群中主节点和哨兵之间的部分网络出现异常（网络不通/网络拥塞）时，可能出现脑裂情况，即哨兵认为主节点挂了，开始推举出新主节点，但旧主节点对外依旧服务着，多个主节点共同运行会导致写入数据一致性出现问题
